## Introduction

As a once and future player of tabletop games, I found early on that what I enjoyed most about such games was their openness - unlike most computer games, the end user can not only open things up and see how they tick, they can create their own content and mechanics. I often found this to be as or more rewarding than the actual game play, which in practice often suffers from crippling pacing and balance issues.

Put these things together - the openness of the system(s) and the issues that can become a barrier to fun gaming, and I have the same thought as any engineer when confronted with a system someone else designed: "Hey, I can do this better!"

### Doing It Better

I've spent many hours in the past building my own game system, and yet none of them has ever gotten off of the proverbial drawing board. So. what is different about this time around? Short version, I'm moving from a waterfall approach to a new, iterated design plan.

The plan this time is to build the “minimal functional version” as a prototype of sorts, covering an internally-consistent subset of the core mechanics. And then I'm going to run a few games with it. This has two purposes: one to generate feedback and actual play experience during the development process, both of which are invaluable. And second, to maintain enthusiasm for the project throughout the process by experiencing the game in action.

### Pipes and Plumbers

So, having set myself the task of developing a new game system, I have decided that the first iteration will be a basic combat system set in the world of a certain world-famous, turtle-stomping plumber. This project will be titled, provisionally, Pipes and Plumbers in the age-old RPG naming tradition (for those of you who favor White Wolf, you can call it Mustache: the Mushrooming instead).
